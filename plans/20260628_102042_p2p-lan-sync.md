# P2P LAN Sync (offline device-to-device account transfer)

**Status:** completed

Port of the SMS Sentry `P2PSyncEngine.kt` design (`p2p-sync-implementation-guide.md`) to this
Flutter authenticator app: offline, LAN-based, two-device sync secured by an out-of-band
high-entropy pairing code with AES-GCM authenticated encryption at the payload layer.

---

## 1. What the issue / request is

The user wants device-to-device account+group transfer over the local network, without a server,
account, or cloud, using the guide's model: a per-session ~320-bit pairing code shown on the host
and typed into the client out-of-band; the code never crosses the wire; the payload is sealed with a
PBKDF2-derived AES-256-GCM key so a wrong code fails GCM tag verification (authentication = decrypt).

### The conflict this introduces (must be resolved as part of the work)

This app is **documented as fully offline** and sync is an explicit **non-goal**:

- `docs/architecture.md` §2 (Non-Goals): "Cloud sync, server-side account storage, or remote
  account recovery." §7/§19: fully offline, no release `INTERNET` permission.
- `docs/security.md` threat model and storage/logging rules are written for an offline app.
- `CLAUDE.md`: architecture.md and security.md are authoritative and win conflicts.

Therefore this feature is a deliberate, conscious reversal of a core design decision. The plan below
treats the documentation + threat-model updates as **first-class deliverables**, not afterthoughts.
P2P sync here is **strictly LAN-only, user-initiated, foreground-only** — it does NOT introduce any
app backend, cloud client, or background networking, which keeps the spirit of "offline" (no
internet dependency) while adding opt-in local transfer.

## 2. Design decisions (Flutter adaptation of the Kotlin guide)

1. **Reuse the existing crypto stack — do not introduce new primitives.**
   - PBKDF2-HMAC-SHA256 via `pointycastle` (already used in `export_import_service.dart` `_deriveKey`).
   - AES-256-GCM via the `encrypt` package (already used app-wide).
   - `Random.secure()` / `encrypt.SecureRandom` for salt, IV, and pairing code.
   - Keep this app's **300,000** PBKDF2 iterations (`AppConstants.pbkdf2Iterations`), not the
     guide's 120k.

2. **Reuse the existing backup payload format and import funnel.**
   - Serialize the payload as the existing backup JSON produced by `_dataToJson` (version, created,
     accounts[], groups[]) so the over-the-wire payload is identical to a file backup.
   - On the receiver, route everything through `AccountProvider.importData(...)`, the single
     ingestion funnel, which already dedups (groups by name; accounts by name|issuer|type) and
     **re-encrypts every secret under the receiving device's own key before insert**.

3. **Secret handling (matches `docs/security.md`).** Sender: decrypt secret with device key →
   re-encrypt under the per-session pairing key for transit only. Receiver: decrypt with pairing
   key → `importData` re-encrypts under its device key. Plaintext secrets exist only transiently in
   memory; nothing decrypted is ever logged or written to disk.

4. **Transport.** Plain `dart:io` `ServerSocket`/`Socket`, line-based (`\n`-terminated). No TLS —
   safe only because the code never crosses the wire and the payload is GCM-sealed (guide §10 caveat
   documented in security.md).
   - **Random ephemeral port per session.** The host binds with `ServerSocket.bind(addr, 0)`, lets
     the OS assign a free port, reads it back via `serverSocket.port`, and displays it to the user
     alongside IP + pairing code. The client UI gains a **port field** (the user types IP + port +
     code). This avoids fixed-port conflicts and is mild defense-in-depth; it is **not** a real
     security boundary (a LAN attacker can port-scan), and security.md will say so explicitly.

5. **Host auto-stop on idle (configurable).** When hosting, start a timer; if no client completes a
   successful handshake (valid `HELLO_SYNC`) within the window, the host **automatically tears down
   the `ServerSocket`** and transitions to Idle (or a "timed out" Error). The window is a new
   user-configurable setting (default proposed **120 s**, bounds e.g. 30 s–600 s) persisted via
   `SettingsProvider`/`SharedPreferences`. The timer is cancelled once a transaction authenticates,
   and reset/stopped when the user manually stops hosting or leaves the screen.

6. **Hostile-peer hardening (guide §4) is mandatory, not optional.**
   - Bounded line reader (cap ~4 KB handshake / ~16 MB payload — authenticator payloads are tiny;
     16 MB is generous) instead of an unbounded read-until-newline.
   - Socket read timeout (30 s) and client connect timeout (6 s).
   - Payload caps before ingestion: max accounts (e.g. 5,000), max groups (e.g. 1,000), per-field
     length cap (e.g. 4,096 chars), schema/type checks. Applied before `importData`.

7. **Directionality.** Phase 1 = **one-directional** (host pushes its accounts+groups → client
   imports), matching the guide's base flow and the existing one-way import semantics. Two-way merge
   is explicitly out of scope for this plan (listed as a follow-up).

8. **State model.** A `SyncProvider extends ChangeNotifier` exposing a `SyncState` (sealed-like
   class hierarchy: Idle, Hosting{ip,port,code}, Connecting, Syncing, Completed{imported}, Error{msg}),
   consistent with the app's Provider pattern. No new state-management dependency.

9. **Platform scope.** Android is the primary, fully-supported target. iOS/Windows: engine compiles
   (dart:io is cross-platform) but is gated/untested; iOS local-network entitlement work is a
   follow-up, not in this plan. Document this clearly.

## 4. Wire protocol (as it will be implemented)

```
HOST                                        CLIENT
  bind(addr, port=0) -> OS assigns port
  display IP + port + code; start idle timer (configurable)
  ServerSocket.accept (or auto-stop if idle timer fires first)
  salt = secureRandom(16)                    connect(IP, port), 6s timeout
  -- send base64(salt) cleartext -------->  derive key = PBKDF2(code, salt, 300k)
  derive key = PBKDF2(code, salt, 300k)
                                            -- send encrypt("HELLO_SYNC") -->
  decrypt == "HELLO_SYNC"?  (cancel idle timer on success)
     no  -> send encrypt("DENIED"), abort
     yes v
  -- send encrypt("ACCEPT_SYNC") -------->  decrypt == "ACCEPT_SYNC"? else abort
  payload = _dataToJson(accounts, groups)   (secrets decrypted w/ device key)
  -- send encrypt(payload) -------------->  decrypt -> validate caps -> importData()
```

- Salt in clear is fine (unique-per-session, not secret). Port is not secret either (scannable).
- A valid `HELLO_SYNC` ciphertext IS the authentication; no plaintext password comparison.
- No fallback cipher; any crypto failure surfaces as an error (never a downgrade).
- If the idle timer fires before a successful `HELLO_SYNC`, the host closes the `ServerSocket` and
  goes Idle/timed-out without ever exposing the listener further.

## 5. Testing plan

- `flutter test` must pass after every change (per CLAUDE.md).
- New unit tests as listed in §3; reuse existing platform-channel mock pattern for secure storage
  and `sqflite_common_ffi` for DB-backed provider tests.
- Unit-test the host idle-timeout: server auto-stops when no handshake arrives in the window, and
  the timer is cancelled on successful auth; settings persistence + bounds clamping for the timeout.
- Manual device validation (two devices on one Wi-Fi/hotspot): happy path, wrong code, code typo,
  wrong/changed port, client disconnect mid-transfer, host idle auto-stop, duplicate accounts
  skipped, large account list within caps.
- `flutter analyze` → zero new issues; `dart format`.

## 6. Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| Adds a network attack surface to a previously offline app | LAN-only, foreground, user-initiated; payload GCM-sealed under a 320-bit per-session code; hostile-peer caps/timeouts; **random port + configurable host auto-stop** minimise the listener's exposure window; documented in security.md |
| `INTERNET` permission now in release manifest | Explicitly documented + justified in architecture.md/security.md; no app backend/cloud client added |
| Wrong-code / MITM | Per-session high-entropy code never on the wire; wrong key → GCM tag failure → abort |
| OOM / DoS from malicious peer | Bounded line reader, payload caps, socket timeouts |
| iOS local-network entitlement / Android 13 nearby-wifi nuances | Android-first; iOS/Windows gated + flagged as follow-up |
| Secret leakage | Plaintext only transient in memory; never logged (debug or release); reuse import funnel |

## 7. Out of scope (follow-ups)
